# 工作日志转工作月报系统技术方案

## 1. 项目概述

### 1.1 背景
基于现有 TraceFlow 工作日志系统，实现利用大语言模型能力，自动将用户日常记录的工作日志转换为结构化、专业化的月度工作报告。

### 1.2 目标
- 自动提取工作日志中的关键工作任务
- 智能分析工作内容并进行分类统计
- 生成符合企业规范的工作月报
- 支持个性化模板和自定义内容

## 3. 核心模块设计

### 3.1 数据采集与预处理模块

#### 功能描述
负责从后端获取工作日志数据，并进行格式标准化、数据清洗、结构化转换，为后续大模型处理做准备。

#### 主要功能
- **日志获取与筛选**
  - 按用户、时间范围获取日志
  - 支持自定义日期区间查询
  
- **格式标准化**
  ```typescript
  interface StandardLog {
    logId: string;
    userId: string;
    userName: string;
    date: string;               // ISO 8601
    workItems: WorkItem[];
    tags: string[];
    deliverables: Deliverable[];
    content: string;
  }
  
  interface WorkItem {
    category: string;
    description: string;
    duration?: number;          // 预估工时，可选
    files?: File[];
  }
  ```
  
- **数据清洗**
  - 去除无效/空白日志
  - 规范化日期格式
  - 统一标签命名
  - 敏感信息过滤（可选）

#### 处理流程
```
原始日志 → 格式校验 → 字段补全 → 内容清洗 → 结构化数据
```

#### 接口调用规范

**统一的 LLM 接口封装**
```typescript
interface LLMConfig {
  provider: 'openai' | 'anthropic' | 'deepseek' | 'qwen';
  model: string;
  apiKey: string;
  temperature?: number;        // 默认为0.3，降低创意性增加稳定性
  maxTokens?: number;
  timeout?: number;            // 超时时间，默认 60s
}

interface LLMRequest {
  systemPrompt: string;
  userPrompt: string;
  context?: string;           // 附加上下文信息
  responseFormat?: 'json' | 'text';  // 输出格式
}

interface LLMResponse {
  success: boolean;
  content?: string;
  data?: any;                  // 当 responseFormat 为 json 时
  tokensUsed?: {
    prompt: number;
    completion: number;
  };
  error?: string;
}

class LLMService {
  constructor(config: LLMConfig);
  async call(request: LLMRequest): Promise<LLMResponse>;
  async callWithRetry(request: LLMRequest, maxRetries = 3): Promise<LLMResponse>;
}
```

#### 参数优化策略

| 参数 | 建议值 | 说明 |
|------|--------|------|
| temperature | 0.3-0.5 | 较低的温度让输出更稳定 |
| top_p | 0.9 | 适当增加多样性但不过度 |
| max_tokens | 2000-4000 | 根据月报长度调整 |
| presence_penalty | 0 | 不需要 |
| frequency_penalty | 0 | 不需要 |

```typescript
async function aggregateUserLogs(
  userId: string,
  startDate: Date,
  endDate: Date
): Promise<StandardLog[]> {
  // 1. 从数据库获取日志
  const logs = await kv.getByPrefix('log:');
  
  // 2. 过滤用户和时间
  const userLogs = logs.filter(log => {
    return log.userId === userId &&
           new Date(log.dateTime) >= startDate &&
           new Date(log.dateTime) <= endDate;
  });
  
  // 3. 标准化格式
  return userLogs.map(normalizeLog);
}
```

## 5. 错误处理机制

### 5.1 常见错误类型

| 错误类型 | 场景 | 处理策略 |
|----------|------|----------|
| API 超时 | 大模型响应慢 | 重试 + 降级到小模型 |
| Token 超限 | 日志内容过长 | 智能摘要 + 分批处理 |
| 格式错误 | JSON 解析失败 | 重试 + 格式修正提示 |
| 内容敏感 | 输出含违规内容 | 内容审核 + 重新生成 |
| 网络错误 | 请求失败 | 重试 + 降级缓存 |

### 5.2 重试策略

**指数退避重试算法：**
```typescript
async function retryWithBackoff<T>(
  fn: () => Promise<T>,
  maxRetries = 3,
  baseDelay = 1000
): Promise<T> {
  let lastError: Error;
  
  for (let i = 0; i < maxRetries; i++) {
    try {
      return await fn();
    } catch (error) {
      lastError = error;
      await new Promise(resolve => 
        setTimeout(resolve, baseDelay * Math.pow(2, i))
      );
    }
  }
  
  throw lastError;
}
```

### 5.3 降级方案

- **降级 1**: 大模型不可用 → 使用统计数据模板
- **降级 2**: 高级分析失败 → 只做基础统计
- **降级 3**: 生成失败 → 返回历史成功的月报

## 6. 性能优化策略

### 6.1 缓存策略

**多级缓存架构：**
```typescript
class ReportCache {
  // 1. 内存缓存（最快，短期）
  private memoryCache: Map<string, CachedReport>;
  
  // 2. KV 存储缓存（中期，已生成的月报）
  // 3. 冷存储（历史月报归档）
  
  async get(userId: string, year: number, month: number): Promise<CachedReport | null>;
  async set(userId: string, year: number, month: number, report: MonthlyReport): Promise<void>;
```

**缓存失效策略：**
- 用户新增/编辑日志 → 相关月报缓存失效
- 月报模板更新 → 重新生成

- 批量处理，减少 API 调用次数
- 压缩上下文，减少 token 使用
- 使用流式输出提升体验
